## Context

This skill performs a structured readability review of a documentation file in `docs/`. The review evaluates the document as a reader would experience it -- reading top to bottom -- and identifies structural, content, and consistency issues.

## Your Task

1. **Identify the target file.** The user will specify a doc file (e.g., `docs/Value-Objects.md`). If not specified, ask.

2. **Read the full target file** and any related files in the same doc family (e.g., if reviewing `Value-Objects.md`, also skim `Value-Objects-Customization.md` and `Value-Objects-Framework-Integration.md` for cross-reference accuracy).

3. **Review for these categories:**

   **Structure & Flow**
   - Can a reader follow the document top to bottom without encountering unexplained concepts?
   - Are sections in logical order (concepts before examples, simple before advanced)?
   - Is the Table of Contents complete and synchronized with actual headings?
   - Are there orphaned sections not in the ToC?

   **Content Accuracy**
   - Do code examples match the actual library behavior? (Use Serena to verify against source code when in doubt)
   - Are cross-references to other doc pages correct? (Check anchor links match headings)
   - Are there misleading statements or outdated information?

   **Redundancy & Gaps**
   - Are there sections that repeat content from other sections?
   - Are there pointer-only sections ("See X for details") that could be inline notes instead?
   - Are there important features mentioned without introduction?

   **Readability**
   - Are technical explanations accessible to the target audience?
   - Are code examples minimal and focused?
   - Is formatting consistent (header casing, blockquote usage, code block style)?

   **Consistency with Sibling Docs**
   - Does this page follow the same patterns as other doc pages in the same family?
   - Are example types/names consistent across pages?

4. **Present findings** in a structured format:

   ```
   ## Review: [filename]

   ### Findings

   | # | Issue | Severity | Category | Proposed Fix |
   |---|-------|----------|----------|--------------|
   | 1 | ...   | High     | Flow     | ...          |
   | 2 | ...   | Medium   | Content  | ...          |
   ```

5. **Ask the user** which findings to address, then enter plan mode if they want changes.

## Guardrails

- Do NOT rewrite prose to sound more "professional" -- the user has explicitly flagged AI-sounding text as undesirable. Keep language natural and direct.
- Do NOT add unnecessary generalizations like "In modern software..." or "You often need to..."
- When proposing text changes, keep them concise. A simple intro sentence is better than a paragraph.
- Simplify technical language in user-facing docs. Avoid constructions like "2+ distinct non-stateless reference types" -- rephrase for human readers.
- Preserve the existing doc conventions: `Console.WriteLine` in examples (not logger injection), consistent example items across pages.
- Do NOT touch frozen/archived docs in `docs/version-7/` or `docs/version-8/`.
